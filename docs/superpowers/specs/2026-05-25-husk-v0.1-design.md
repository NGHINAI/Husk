# Husk v0.1 — Architectural Spec

**Status:** Draft
**Author voice:** the AI agent that has to actually use this
**Scope:** the redesign that takes Husk from "browser engine wrapped in tools" to "model of the web that agents reason against"
**Time horizon:** ~6-8 weeks
**Foundation:** everything through v0.0.16-m17 + post-M17 patches is intact and becomes the substrate

---

## 0. Preamble — why this exists

Husk v0 (M1 through M17) shipped a working AI browser engine. 21 MCP tools, real Chrome adapter, seamless handoff, snapshot maximalism, dynamic workflows, multi-context HITL. It works on simple sites and the demo-day sites: Wikipedia, Hacker News, basic forms.

It does NOT work cleanly on LinkedIn, Gmail, Salesforce, modern GitHub, Notion, Linear, or any site where the agent needs to **reason about state** rather than just emit clicks.

Every time we hit a wall (BroadcastChannel polyfill gap, opened-modal half-done, bot detection refusal, OTP escalation, engine routing missing), we patched it. The patches work. The patches stack up. Each patch is two more sentences in an MCP description and a watchdog rule.

This document is the answer to: **what would we build if we started over, knowing what we know?**

Short version: not a browser. A **model of the web that agents reason against**, with the actual browsing as a substrate the agent never touches directly.

---

## 1. First principles

These five claims are the spec. Everything below follows.

### 1.1 Agents are not humans

A human uses a browser through pixels + mouse + keyboard. An agent uses a browser through tokens + intentions + decisions. Designing for the agent means designing for the second flow, not pretending the agent is a fast human.

Implication: drop the DOM, drop the AX tree, drop the click/type/scroll vocabulary. Those are pixel-substitutes invented because we didn't know how else to expose a page to a non-human.

### 1.2 Agents think in intentions, not events

A human's mental model: "I'll move the mouse here, click this button, type these characters, press Enter."

An agent's mental model: "Send a connect request to Vikash Singh."

Husk v0 forces the agent to translate the second into the first. Husk v0.1 takes intentions directly and translates them internally. The browser does the click/type/wait/verify dance; the agent never sees it unless it fails.

### 1.3 Agents have context limits

Every token shown to the agent is a token taken from its working memory. A 2,000-node AX tree is 50KB. A 200-node terse tree is 5KB. A "you're on the LinkedIn home feed, 3 affordances available" sentence is 30 bytes.

Snapshot Maximalism (M14) was the right instinct executed wrong. The browser should give the agent **what it needs** for the current intention, not everything the page contains.

### 1.4 Agents must verify outcomes — but should not have to

If the agent issues "send connect request" and the browser says `ok: true`, the agent should be able to trust that the connect request was sent. Not "the click landed on a button" — the actual request was sent and acknowledged by LinkedIn's backend.

Verification is the browser's job, not the agent's. The browser sees the page; the agent doesn't. The browser knows what the network requested and what came back; the agent doesn't. Asking the agent to verify is asking the wrong party to do the work.

### 1.5 Agents accumulate experience — the browser should remember

A human learns LinkedIn's UI once and remembers forever. An agent today re-learns LinkedIn every session because the orchestrator is amnesiac (M4 site-graph stores selectors but not state machines).

The browser should remember everything: per-site state graphs, action sequences that worked, action sequences that failed, the user's preferences. Each session benefits from every prior session by any agent.

## 2. The framing shift — pages are state machines

The single most important change.

### 2.1 What we have today (v0)

```
Agent: husk_snapshot() → 2000-node AX tree
Agent: parse tree → find login button
Agent: husk_click({intent: "sign in button"}) → ok
Agent: husk_snapshot() → 2000-node AX tree (95% same)
Agent: parse tree → find email field
... 6 more rounds ...
Agent: husk_extract({selectors}) → "Logged in as Nirmal"
Agent: maybe? presumably?
```

Cost: 12 tool calls, 6 round-trips of context bloat, no verification.

### 2.2 What v0.1 looks like

```
Agent: intend("login", {site: "linkedin.com", email, password})
Husk:  [state: login_form] → [transition: submit] → [state: 2fa_prompt]
Husk:  [needs_human: provide_2fa_code]
Agent: relay to user → user provides code
Agent: provide("2fa_code", "638271")
Husk:  [transition: submit_code] → [state: home_feed]
Husk:  → outcome { ok: true, state: "home_feed", evidence: "li_at cookie issued; HTTP 200 on /voyager/api/me; profile name matches input.email" }
Agent: done
```

Cost: 3 tool calls including the human relay. No tree parsing. No selector-by-intent. The agent expresses intent; the browser executes.

### 2.3 The state-machine abstraction

Every site Husk has driven (or learned about from any agent) has a **state graph**:

```yaml
linkedin.com:
  states:
    visitor:                                # not logged in
      affordances: [login, search_jobs_public, view_company_page]
      identify_by: "url:^https://www.linkedin.com/?$ + no_session_cookie"
    login_form:
      affordances: [submit_login(email, password)]
      identify_by: "url contains /login + input[name=session_key] present"
    2fa_prompt:
      affordances: [submit_2fa_code(code), use_authenticator_app]
      identify_by: "url contains /checkpoint OR aria heading contains '2-step'"
    home_feed:
      affordances: [search, navigate_profile, view_inbox, send_post]
      identify_by: "url=linkedin.com/feed + nav menu present"
    profile_page:
      affordances: [send_connect(note?), send_message, view_full_profile]
      identify_by: "url matches /in/[\\w-]+ + has 'Connect' or 'Message' button"
    connect_modal:
      affordances: [send_without_note, add_note_then_send, cancel]
      identify_by: "dialog role visible + heading contains 'invitation'"
    bot_challenge:
      affordances: [escalate_to_human]   # not solvable by agent
      identify_by: "page text matches /unusual activity|please verify/i OR captcha visible"
  
  transitions:
    visitor → login_form:                  via "click_link(name='Sign in')"
    login_form → home_feed:                via "submit_login" if creds valid
    login_form → 2fa_prompt:               via "submit_login" if 2FA enabled
    login_form → bot_challenge:            via "submit_login" if site flags
    home_feed → profile_page:              via "navigate(url=linkedin.com/in/X)"
    profile_page → connect_modal:          via "click(button='Connect')"
    connect_modal → profile_page+pending:  via "click(button='Send without a note')"
  
  intentions:
    send_connect(person):
      requires_state: profile_page(person=person)
      steps:
        - transition: profile_page → connect_modal
        - transition: connect_modal → profile_page+pending
      verify:
        - network: POST /voyager/api/relationships/sentInvitationViewsV2 returned 201
        - OR ax_state: "Pending" badge appears on profile button
      failure_modes:
        - rate_limited:   network 429 OR text "you've reached the limit"
        - already_connected: text "1st" badge visible
        - blocked_by_target: text "Out of Network"
        - bot_challenge: state transitions to bot_challenge
```

This YAML is the **per-site state graph**. Husk has one per site. It's seeded by the first agent to visit. It's refined by every subsequent agent.

When an agent calls `intend("send_connect", {person})`, Husk:

1. Looks up `linkedin.com` state graph
2. Navigates to `profile_page(person)` (might be multiple steps: home_feed → profile_search → profile_page)
3. Executes the `send_connect` intention's `steps`
4. Runs all `verify` checks
5. If any `failure_modes` pattern matches, returns the structured failure
6. Otherwise returns `{ok: true, state_after, evidence}`

The agent never sees the modal. The agent never has to know LinkedIn has a "Add a note?" dialog. The modal handling is encoded in the state graph **once** by the first agent that hit it.

### 2.4 How the state graph is built

Two ways, hybrid in practice:

**Cold start (exploration mode):** Agent visits a new site. Husk has no state graph. Husk emits a "discovery" snapshot — a richer one with hints. Agent runs intentions, but each step is observed and saved. After N successful traversals, Husk has enough to encode the state graph and switch this site to "execution mode".

**Warm execution (production mode):** Husk has the state graph. Agent's intentions get compiled to action sequences. Snapshots are minimal — just enough to identify current state.

Switching between modes happens per-site, automatically, based on confidence: every transition has a `confidence_score` based on how many times it's worked. Below 0.7 confidence → exploration mode. Above 0.9 → execution mode. Between → execution with verification.

**Communal learning:** state graphs are versioned and shareable. If Agent A successfully maps LinkedIn's connect flow, Agent B inherits it. The graphs live in a per-site SQLite DB (extension of M4's site-graph) and can optionally sync to a community registry (`registry.husk.run/sites/linkedin.com`) if the user opts in.

### 2.5 Identification heuristics

A state's `identify_by` is a small predicate language:

```
url:^https://...
url contains /substring
ax_role:button name:"Sign in" present
ax_text matches /unusual activity/i
network_recent contains url:/voyager/.../api/me status:200
forms[0].fields[?type=password]
cookies contains li_at
title equals "Feed | LinkedIn"
```

Combined with `AND`, `OR`, `NOT`. Husk's task is to evaluate these against the current page snapshot in <5ms. The predicates double as `verify` checks.

```

║  -------------------------------------------     ║

║  • Outcome verifier (predicates + evidence)      ║
║  • Memory: playbooks, preferences, failures      ║
║  • Subscription bus (push events to agent)       ║

║  • Engine fingerprinting (which engine is fast   ║

║  • Per-engine pool (memory-aware, M9/M17 reuse)  ║

                       │

            │   AGENT PROTOCOL    │  (streaming, bidirectional)

                       │

```

Each layer is replaceable. The cognition layer can be extracted and run independently against any browser substrate that speaks CDP. The capability router can be swapped for a manual config. v0 is the bottom and isn't going anywhere.

## 4. The cognition layer in detail

### 4.1 The intention vocabulary

Intentions are the agent's primary verb. Three categories:

**Navigational intentions** — get me to a state
- `nav(url)` — explicit URL
- `goto_state(site, state)` — semantic ("get me to LinkedIn home feed")
- `back()`, `forward()`, `reload()`
- `close_session()`

**Active intentions** — make a state change
- `intend(name, args)` — invokes a named intention from the site's state graph
- `extract(field|fields)` — read structured data
- `wait_for(state)` — block until a state is reached

**Communicative intentions** — talk to the human
- `ask_human(question, options?)` — surface question to user, wait
- `notify(message)` — fire-and-forget notification
- `escalate(reason)` — explicit "I'm stuck"

That's the entire agent-facing surface. ~10 verbs. No `husk_click`, `husk_type`, `husk_scroll`, `husk_snapshot`, `husk_extract`. Those still exist in the substrate but are *implementation details* the cognition layer uses.

### 4.2 The intention compiler

When an agent calls `intend("send_connect", {person})`:

```
1. Resolve current state: snapshot → match against state graph
   → current_state = "search_results"
2. Resolve target state: intention's `requires_state`
   → required_state = "profile_page(person)"
3. Plan transition path: BFS over state graph
   → path = [search_results → profile_page]
4. Execute each transition's action sequence:
   - For each: click/type/scroll/wait/navigate
   - After each: verify the expected post-state
   - On unexpected state: replan from new state, or fail
5. At target state, execute intention's steps:
   - profile_page → connect_modal: click(button='Connect')
   - verify: state machine identified connect_modal
   - connect_modal → profile_page+pending: click(button='Send without a note')
   - verify: state machine identified profile_page+pending
6. Run final verification:
   - network: POST /voyager/... 201? yes
   - ax_state: "Pending" badge? yes
7. Return: {ok: true, state_after: "profile_page+pending", evidence: {...}}
```

The agent saw one call. Six clicks and three navigations happened. None surfaced.

### 4.3 The outcome envelope

Every intention returns:

```typescript
interface Outcome<T = unknown> {
  ok: boolean;
  intention: string;          // "send_connect"
  args: unknown;              // {person: "Vikash Singh"}
  state_before: StateId;      // "search_results"
  state_after?: StateId;      // "search_results_with_pending_invite"
  result?: T;                 // intention-specific payload (e.g., extracted data)
  evidence: Evidence[];       // every verify check that ran + its result
  duration_ms: number;
  reason?: FailureMode;       // when ok:false
  reason_detail?: string;
  recovery_options?: Array<{  // suggested next steps when failed
    label: string;
    intention?: string;
    needs_human?: boolean;
  }>;
  steps_observed: TransitionLog[];  // for debugging — the action sequence executed
}

interface Evidence {
  predicate: string;          // "network: POST /voyager/.../sentInvitationViewsV2 returned 201"
  passed: boolean;
  observed_value?: unknown;
}
```

The agent has a clear contract: `ok:true` means **every verify check passed**, with `evidence` to inspect. `ok:false` means something specific went wrong, with a `reason` from a finite taxonomy.

### 4.4 The failure-mode taxonomy

A finite vocabulary of why intentions fail. Roughly 30 reasons, grouped:

**State-machine reasons:**
- `unknown_site` — no state graph for this domain
- `unknown_state` — current state doesn't match anything in the graph
- `no_path_to_target` — graph has no route from current to required state
- `state_drift_mid_execution` — site moved to an unexpected state during the action sequence

**Network reasons:**
- `network_failure`, `network_timeout`, `network_throttled`

**Site-side reasons:**
- `rate_limited`, `account_locked`, `bot_challenge`, `2fa_required`, `permission_denied`
- `content_not_found`, `feature_unavailable`

**Human reasons:**
- `needs_human` — explicit handoff
- `needs_credentials`, `needs_2fa_code`, `needs_payment_confirmation`
- `human_declined`, `human_timeout`

**Engine reasons:**
- `engine_unsupported` (M17), `engine_crashed`, `out_of_memory`, `pool_exhausted`

Each reason has documented recovery strategies. `rate_limited` → wait + retry with exponential backoff (caller can opt). `bot_challenge` → escalate to seamless handoff. `needs_2fa_code` → escalate to `ask_human`.

The cognition layer handles **automatic** recovery for some; surfaces others to the agent. The agent's contract is: "if `ok:false`, look at `reason` + `recovery_options`."

### 4.5 The subscription bus

Agents subscribe to events:

```
agent.subscribe("state_change", session_id=*)
agent.subscribe("captcha_detected", session_id=current)
agent.subscribe("rate_limit_approaching", site="linkedin.com")
agent.subscribe("new_message", site="gmail.com", folder="inbox")
agent.subscribe("network_idle", session_id=current, debounce=500ms)
```

Events come via the streaming protocol (§7). Agents don't poll. The cognition layer knows when state changes; the agent gets told.

Pre-built subscriptions for common signals:
- `state_change` — current site state changed
- `network_complete` — all in-flight requests settled
- `error_appeared` — error banner / dialog showed up
- `user_intervention_required` — a state we know agents can't handle alone
- `rate_limit_window_reset` — rate-limit cooldown expired
- `new_data` — for sites where polling for updates matters (Gmail, Slack, Discord, etc.)

### 4.6 The confidence engine

Every state, transition, and verify check has a **confidence score** (0.0–1.0) based on observation history.

- New state graph: all confidences = 0.5 (neutral prior)
- Successful traversal: +0.05 per step (capped at 0.99)
- Failed traversal at this step: -0.10 (floor 0.05)
- Long time since last observation: gradual decay

When the agent issues an intention:

- **High confidence (>0.9):** execute directly, minimal verification overhead
- **Medium (0.7–0.9):** execute, verify each step
- **Low (<0.7):** drop to exploration mode — snapshot more aggressively, log everything for the state graph to learn

This means the first agent to use LinkedIn pays the exploration tax (slower, more snapshots). Every subsequent agent rides the learned graph in execution mode.

### 4.7 Memory model

Per-site SQLite DB, extending M4's site-graph. Tables:

```sql
states (site, state_id, identify_predicate, confidence, last_seen, observed_count)
transitions (site, from_state, to_state, action_sequence, confidence, success_count, failure_count, avg_duration_ms)
intentions (site, name, args_schema, requires_state, steps_yaml, verify_yaml, failure_modes_yaml)
playbooks (site, name, intention_chain_yaml, success_count, last_used)
preferences (site, key, value, scope)      -- e.g. "linkedin.com / never_auto_post=true"
failures (site, intention, reason, frequency, last_seen, escalation_strategy)
user_profile (site, user_identity_hash, learned_quirks_yaml)
```

Memory persists across orchestrator restarts. Memory syncs across local machines via the user's vault (optional encrypted sync to a self-hosted registry).

### 4.8 Playbooks (compound intentions)

A playbook is a named compound intention the agent can invoke. Example:

```yaml
name: send_connects_to_list
site: linkedin.com
inputs:
  - people: list[string]    # LinkedIn slugs or names
  - rate_limit_pause_s: int = 30
  - max_pause_on_block: int = 300
steps:
  - foreach person in people:
      - try:
          - intend: send_connect(person)
          - sleep: rate_limit_pause_s
        catch rate_limited:
          - sleep: max_pause_on_block
          - retry
        catch bot_challenge:
          - escalate: needs_human (reason: linkedin_bot_challenge)
          - resume: after_user_unblocks
output:
  - sent: list[person where ok=true]
  - failed: list[{person, reason}]
```

Playbooks are first-class. Agents invoke them: `play("send_connects_to_list", {people: [...]})`. Husk records playbook success/failure rates and surfaces the best playbook for a given intention class.

Playbooks are also community-shareable. If your team builds a great "schedule_calendar_meeting" playbook for Google Calendar, you can publish it.

## 5. The capability router

Generalizes M17's engine selection.

### 5.1 The shift

Today: `husk_create_session({engine: "lightpanda" | "chrome" | "auto"})`. Agent has to know engines exist.

v0.1: `session({requires: [...capabilities]})`. Agent declares what the task needs. Router picks the engine.

### 5.2 Capability vocabulary

```typescript
type Capability =
  | "javascript"            // executes JS at all
  | "modern_javascript"     // BroadcastChannel, IndexedDB, ServiceWorker, etc.
  | "cookies_persistent"    // cookies persist across page loads
  | "real_user_session"     // sign-in state from user's actual browser
  | "screenshot"            // can render to PNG
  | "video_recording"       // can record session video (rare)
  | "file_upload"           // can upload local files
  | "file_download"         // can download files
  | "multi_tab"             // can have multiple tabs sharing state
  | "permissions"           // can grant geolocation/notifications etc.
  | "shadow_dom"            // can traverse web component shadow DOM
  | "iframe_cross_origin"   // can read cross-origin iframes
  | "extension_access"      // can read state from user's Chrome extensions
  | "fast"                  // sub-100ms cold start
  | "low_memory"            // <100MB per session
  | "stealth"               // bot-detection evasion
```

### 5.3 Routing rules

The router picks the cheapest engine that satisfies all required capabilities. Order:

1. `lightpanda` — `javascript`, `cookies_persistent`, `fast`, `low_memory`. Default for everything that doesn't need more.
2. `lightpanda_stealth` (planned) — same + `stealth`. For scraping without user session.
3. `chrome_headless` — `modern_javascript`, `screenshot`, `file_upload`, `file_download`, `multi_tab`, `shadow_dom`. ~500MB, 1.5s cold.
4. `chrome_headed_isolated` — Chrome with isolated profile. Adds `permissions`. For when you need to see a real-looking browser.
5. `chrome_user_profile` — user's real Chrome with their cookies/history. Adds `real_user_session`, `extension_access`. For sites that need pre-existing auth.

If no engine satisfies all required capabilities, the router returns `{ok: false, reason: "no_engine_for_capabilities", missing: [...]}`. The agent can drop a capability, ask the user to install something, etc.

### 5.4 Mid-task swap (generalized)

M17 swaps lightpanda → chrome on page-health failure. v0.1 generalizes:

- Any intention can declare `requires_capability_at_step: [...]`. The router upgrades the engine before that step.
- The cognition layer learns which sites need which engine. Next time, it picks correctly from the start.

Example: a LinkedIn connect flow's `submit_login` step is annotated `requires: real_user_session`. Husk knows: route to `chrome_user_profile` before this step. The seamless handoff (M16) becomes invisible — the router handles it.

### 5.5 Capability-based handoff

Capabilities can include `requires_human_intervention: ["captcha_solving", "2fa_code"]`. When such a capability is required and no automated path exists, the router automatically engages M16's seamless handoff. Again, invisible to the agent.

## 6. The new tool surface

Drastically smaller. Eight tools for the agent. Three are the "do something" verbs, three are the "see / change state" verbs, two are HITL.

| Tool | Purpose | Example |
|---|---|---|
| `husk_session` | Create / close / list sessions | `husk_session({create: {requires: ["modern_javascript", "real_user_session"]}})` |
| `husk_nav` | Get to a URL or named state | `husk_nav({state: "home_feed", site: "linkedin.com"})` |
| `husk_intend` | Issue a named intention | `husk_intend({name: "send_connect", args: {person: "Vikash Singh"}})` |
| `husk_extract` | Read structured data from current state | `husk_extract({fields: ["title", "first_para"]})` |
| `husk_play` | Invoke a compound playbook | `husk_play({name: "send_connects_to_list", inputs: {...}})` |
| `husk_observe` | Get current state + affordances (replaces snapshot) | `husk_observe({verbosity: "minimal" | "rich"})` |
| `husk_ask` | Ask the human something | `husk_ask({question, options?})` |
| `husk_escalate` | Hand off to human | `husk_escalate({reason, context})` |

That's it. **8 tools.** Down from 21.

The 21 v0 tools still exist underneath, accessible via a `husk_advanced` namespace for power users who want to drop down to the click/type/scroll vocabulary. Most agents never touch them.

### 6.1 What disappears

- `husk_create_session`, `husk_close_session` → folded into `husk_session`
- `husk_snapshot`, `husk_snapshot_diff` → replaced by `husk_observe`
- `husk_click`, `husk_type`, `husk_scroll`, `husk_press_key`, `husk_upload` → encapsulated in `husk_intend`
- `husk_wait_for` → encapsulated in `husk_intend` (every intention has implicit wait + verify)
- `husk_extract`, `husk_batch_visit` → folded into `husk_extract` + `husk_play`
- `husk_login`, `husk_credentials_set`, `husk_vault_list_profiles`, `husk_vault_clear` → folded into `husk_intend({name: "login", ...})` or `husk_session({use_profile: "..."})`
- `husk_set_policy` → still exists in `husk_advanced` (rarely used)
- `husk_handoff`, `husk_resume` → folded into `husk_escalate` + automatic handling
- `husk_dialog`, `husk_version` → `husk_advanced`

### 6.2 What `husk_observe` actually returns

In `minimal` mode (default):

```json
{
  "session_id": "s1",
  "site": "linkedin.com",
  "state": "home_feed",
  "state_confidence": 0.94,
  "affordances": [
    {"name": "search", "args": ["query"]},
    {"name": "navigate_profile", "args": ["slug"]},
    {"name": "view_inbox"},
    {"name": "send_post", "args": ["content"]}
  ],
  "session_history": ["nav home", "intend(login)", "current"],
  "watch_url": "..."
}
```

That's it. ~500 bytes. No AX tree. No DOM. The agent has everything it needs to choose the next intention.

In `rich` mode (when explicitly requested):

Everything v0's M14 snapshot returned, including the AX tree, network ring, console, meta, forms, summary, screenshot. For when the agent legitimately needs raw access.

### 6.3 What `husk_intend` actually returns

```json
{
  "ok": true,
  "intention": "send_connect",
  "args": {"person": "Vikash Singh"},
  "state_before": "search_results",
  "state_after": "search_results_with_pending_invite",
  "evidence": [
    {"predicate": "network: POST /voyager/api/relationships/sentInvitationViewsV2 returned 201", "passed": true},
    {"predicate": "ax: 'Pending' badge visible on Vikash's card", "passed": true}
  ],
  "duration_ms": 1240,
  "steps_observed": [
    {"verb": "click", "target": "Invite Vikash Singh to connect", "duration_ms": 320},
    {"verb": "wait_for", "predicate": "dialog appears", "duration_ms": 180},
    {"verb": "click", "target": "Send without a note", "duration_ms": 280},
    {"verb": "wait_for", "predicate": "dialog closes + 'Pending' badge", "duration_ms": 460}
  ]
}
```

The agent can ignore `evidence` and `steps_observed` (they exist for debugging + replay). It just reads `ok` and `state_after`.

## 7. The protocol — streaming, bidirectional

### 7.1 Why streaming

Today's MCP is request/response: one tool call, one result. For multi-step intentions that take 30+ seconds (rate-limit pauses, handoff escalations, captcha waits), the agent has to either block or poll. Blocking ties up the conversation; polling wastes tool calls.

v0.1 uses a streaming protocol. The agent issues an intention. Husk pushes progress events as they happen. The agent receives the final outcome when ready. The agent can issue additional intentions concurrently (different sessions) or cancel an in-flight intention.

### 7.2 The wire protocol

Two viable choices:

**Option A: MCP streaming extension.** When MCP adds streaming tools (in spec discussion), use that. Cleanest for AI-client compatibility.

**Option B: Custom WebSocket / SSE protocol.** Husk runs its own bidirectional channel. The MCP server bridges single-shot tools onto streaming intentions for clients that don't support streaming.

Both work. Implement (B) first because MCP streaming isn't standardized; expose (A)-compatible shim for AI clients.

### 7.3 Message shapes

```jsonl
// Agent → Husk
{"id": 42, "verb": "intend", "args": {"name": "send_connect", "args": {"person": "Vikash Singh"}}}

// Husk → Agent (progress, optional, may be many)
{"id": 42, "event": "progress", "stage": "transitioning", "state": "search_results → profile_page"}
{"id": 42, "event": "progress", "stage": "opened_modal", "state": "connect_modal"}
{"id": 42, "event": "progress", "stage": "verifying", "predicate": "network 201"}

// Husk → Agent (terminal)
{"id": 42, "event": "outcome", "outcome": {"ok": true, "state_after": "...", "evidence": [...]}}

// Agent → Husk (cancel)
{"id": 42, "verb": "cancel"}

// Husk → Agent (independent events, no id)
{"event": "state_change", "session": "s1", "state": "rate_limited"}
{"event": "captcha_detected", "session": "s1", "captcha_type": "recaptcha_v2"}
```

### 7.4 Compound playbook streaming

When `husk_play` runs a long playbook:

```jsonl
{"id": 99, "verb": "play", "args": {"name": "send_connects_to_list", "inputs": {...}}}
{"id": 99, "event": "playbook_started", "name": "send_connects_to_list", "estimated_duration_s": 90}
{"id": 99, "event": "step", "index": 1, "of": 9, "sub_intention": "send_connect", "args": {"person": "Vikash"}, "outcome": {"ok": true}}
{"id": 99, "event": "step", "index": 2, "of": 9, "sub_intention": "send_connect", "args": {"person": "Ibrahim"}, "outcome": {"ok": true}}
{"id": 99, "event": "step", "index": 3, "of": 9, "sub_intention": "send_connect", "args": {"person": "Charles"}, "outcome": {"ok": false, "reason": "rate_limited"}, "pause_s": 30}
{"id": 99, "event": "step", "index": 3, "of": 9, "sub_intention": "send_connect", "args": {"person": "Charles"}, "outcome": {"ok": true}, "retried": true}
...
{"id": 99, "event": "outcome", "outcome": {"ok": true, "result": {"sent": [...], "failed": [...]}}}
```

The agent watches the stream. The user (watching the chat) sees real progress, not a frozen tool call.

### 7.5 Cancellation, pause, resume

Agents can `cancel` an in-flight intention. Husk gracefully stops at the next safe checkpoint. Husk may also `pause` an intention (it's in a transient state) and `resume` later (after a captcha is solved, for example).

## 8. Multi-agent / multi-tab choreography

### 8.1 Multi-tab semantics

M15 introduced tab groups. v0.1 makes them first-class in intentions:

```
husk_session({create: {tabs: 3, share: "cookies"}})
→ {session_id: "s1", tabs: ["t1", "t2", "t3"]}

husk_intend({name: "research_company", args: {company: "Acme"}, tab: "t1"})
husk_intend({name: "research_competitors", args: {of: "Acme"}, tab: "t2"})
husk_intend({name: "find_news", args: {about: "Acme"}, tab: "t3"})
// All three run in parallel
```

Tabs share cookies (so a logged-in session is one login for all three tabs). Each tab has its own state. The agent can wait on all three or any one (`husk_wait_for({tabs: ["t1", "t2"], any})`).

### 8.2 Multi-agent

Multiple agents can share a Husk instance. Each gets a per-agent namespace; sessions are isolated by default but can be shared explicitly.

A common pattern: an "orchestrator agent" delegates subtasks to specialist agents. Each specialist gets a sub-session in the orchestrator's tab group. They communicate via the session memory.

```
husk_session({create: {sub_of: "s_parent", agent: "specialist_linkedin"}})
husk_intend({name: "send_connect", args: {person}, session: "s_specialist"})
```

The cognition layer enforces resource quotas across agents (no one agent monopolizes Chrome processes).

### 8.3 Cross-site reasoning

The browser knows you're logged into LinkedIn AND Gmail AND Stripe. When an agent issues an intention that needs cross-site context ("send me the LinkedIn profile of the person who emailed me about the invoice"), the cognition layer can chain across sites:

```
husk_intend({name: "find_sender_linkedin", args: {email_id: "..."}})
// Internally:
//   1. Gmail: get email → extract sender email + name
//   2. LinkedIn: search by name + company → match
//   3. Return profile URL
```

The agent issues one intention; three sites participated. Cross-site reasoning is opt-in (user permission flow on first such intention per site combo).

## 9. Memory in detail

### 9.1 What gets stored

```yaml
~/.husk/v0.1/memory/
  sites/
    linkedin.com/
      state-graph.yaml           # the per-site model
      transitions.sqlite         # observation history
      playbooks/
        send_connects.yaml
        scrape_company_page.yaml
      failure-log.sqlite         # what's failed + how often
      user-profile.yaml          # learned quirks for this user
    gmail.com/
      ...
  global/
    capabilities.yaml            # engine capability cache
    failure-recovery.yaml        # generic recovery strategies
    user-preferences.yaml        # global prefs ("never auto-post")
  audit/
    actions.jsonl                # every intention executed + outcome
    state-transitions.jsonl      # state-machine learning data
```

### 9.2 Learning loop

Every intention run feeds back into memory:

- Success → transition confidence up, intention's `success_count` up
- Failure → log the failure mode, decrement confidence, suggest new edge case for the state graph
- New state encountered → flag for state-graph review (manual or automated proposal)

Over time, the state graph for any frequently-visited site approaches completeness.

### 9.3 Per-user calibration

Memory remembers user preferences encoded as constraints on intentions:

```yaml
linkedin.com:
  preferences:
    - never_auto_connect_without_note: true   # always pause for note text
    - rate_limit_pause_s: 60                  # be more conservative than default
    - never_visit_profiles_of: ["competitors_list_id_42"]
  user_quirks:
    - profile_url_format: "/in/[name]"        # this user uses old-format URLs
```

Agent issues `send_connect(person)`. Husk reads preferences. If `never_auto_connect_without_note=true`, Husk pauses and asks the user for note text. Surfaces this through `husk_ask` automatically.

### 9.4 Sharing / portability

Memory is portable. Export as a `.huskpack` (tarball + signed manifest). Share with teammates. Sync via the user's cloud storage (Dropbox, S3, etc.) — encrypted with `HUSK_VAULT_KEY`.

Community state graphs (opt-in) sync from `registry.husk.run` for sites people care about. Husk verifies the graph against the live site on first load before trusting community data.

## 10. Safety, trust, and budget

### 10.1 Trust levels per site

Sites get trust scores 0-3:

- **0 (high-risk):** banking, government, payment, healthcare. **Every action confirmed.** No auto-handoff. Watchdog is always strictest. Memory access is read-only by default.
- **1 (sensitive):** auth providers, email, work tools (Salesforce, Workday). Destructive actions confirmed. Memory writeable but with audit log.
- **2 (regular):** most consumer apps (LinkedIn, Twitter, Reddit). Default behavior.
- **3 (low-risk):** news sites, documentation, Wikipedia. Maximum agent autonomy.

Site → trust mapping ships with Husk; user can override.

### 10.2 Destructive action gating

Every intention is marked `destructive: true | false`. Examples of destructive: `send_post`, `send_message`, `submit_payment`, `delete_anything`, `connect_account`, `permanently_change_settings`.

For destructive intentions on trust-level-0-or-1 sites: Husk pauses and asks the user via `husk_ask` regardless of any other config. The user (not the agent) is the final gate.

### 10.3 Budget enforcement

Sessions declare budgets at create-time:

```
husk_session({create: {
  budget: {
    max_intentions: 50,
    max_seconds: 600,
    max_user_touches: 5,
    max_dollars: 10,         // for payment intentions
    rate_limit_max_pause: 300
  }
}})
```

Husk hard-limits at budget exhaustion. Returns `{ok: false, reason: "budget_exhausted"}`. User can extend the budget interactively.

### 10.4 Audit trail

Every intention + transition + state change is logged in `~/.husk/v0.1/memory/audit/`. Tamper-evident (each entry signed; chain via prev-hash). Replayable.

If the user wants to know "what did Husk do on LinkedIn last week?", they get a clean log.

### 10.5 Sandboxing

Sessions are isolated. State graph for `linkedin.com` cannot drive `gmail.com`. Cookies don't leak. Cross-site reasoning is explicit + user-permission-gated.

### 10.6 Network observability

User can see every outbound network request, optionally redact sensitive params (passwords, OTPs). Useful for verifying that Husk isn't exfiltrating data.

## 11. The synthetic browsing path

When a site exposes a clean API behind its UI, Husk SHOULD prefer the API. Examples:

- **GitHub** has a great REST API. Asked to "list my open PRs"? Call `api.github.com/issues?filter=created&state=open` directly. Zero rendering. Instant.
- **Reddit** has `.json` endpoints for every page. Asked to "summarize today's Hacker News equivalents on r/programming"? Hit `reddit.com/r/programming.json` directly.
- **Twitter** has documented APIs (when authenticated). Use them.

The cognition layer keeps a per-site API catalog. When an intention can be served via API:

1. Try the API first (faster, more reliable, no rendering needed)
2. Fall back to the UI flow if the API fails (rate limit, deprecated endpoint, etc.)
3. Compare outputs — if API and UI disagree, flag for review

This is M14's `likely_api_endpoints` taken to its logical conclusion. The cognition layer doesn't just suggest the API to the agent — it uses the API itself, and the agent is none the wiser.

For some sites (LinkedIn especially), the API is closed/internal. Husk can use the same JSON endpoints LinkedIn's React frontend uses, since the cookies are valid for both. This shifts a `send_connect` from "render React page + click button" to "POST `/voyager/api/relationships/sentInvitationViewsV2`" — sub-second.

## 12. Confidence, exploration, calibration

### 12.1 The exploration-execution spectrum

For any site, Husk operates on a spectrum:

- **Cold (no graph):** every action is exploratory. Heavy snapshot. Verbose. Slow. Goal: learn the state graph.
- **Warm (partial graph):** common paths cached. Snapshots minimal. Verify each step. Goal: refine the graph.
- **Hot (complete graph):** intentions execute as compiled action sequences. Minimal verify. Goal: maximum throughput.

The cognition layer manages the spectrum per-site, per-intention. Hot for the LinkedIn-connect-flow (we've done it 100+ times). Warm for LinkedIn-job-application (we've done it 5 times). Cold for some niche enterprise SaaS we just encountered.

### 12.2 The calibration loop

Periodically, Husk runs synthetic verifications: pick a known-working playbook on a known site, run it, confirm outcome. If failure rate climbs, drop confidence for that path. Site might have changed; the graph might need updating.

This is a background process — runs on a small percentage of sessions opportunistically.

### 12.3 The "I'm not sure" surface

When confidence is low (cold start, or a known-flaky path), Husk surfaces uncertainty to the agent:

```json
{
  "ok": true,
  "state_after": "search_results",
  "confidence": 0.62,
  "uncertainty_reasons": ["first_time_on_this_state", "no_explicit_verify_passed"]
}
```

The agent can decide: trust it, re-verify with `husk_observe`, ask the user, etc.

## 13. Time, scheduling, persistence

### 13.1 Scheduled intentions

```
husk_intend({name: "send_connect", args: {person}, at: "2026-05-26T09:00:00-05:00"})
```

Husk schedules. The user's machine (or a hosted Husk instance) wakes up at the time and executes. Reports back via the subscription bus.

### 13.2 Long-running intentions

Some intentions are days-long (monitor a job application status, watch for a Slack message). Husk supports these as background subscriptions:

```
husk_subscribe({
  trigger: {event: "linkedin_application_status_change", application_id: "..."},
  action: {intend: "notify_user", args: {message: "Application status: ${new_status}"}},
  duration_days: 30
})
```

The agent declares it once and disconnects. Husk maintains the watch independently. Notifies via the subscription bus when the trigger fires.

### 13.3 Session resumption

A session can be paused (e.g., due to an unsolved captcha) and resumed days later. The state graph + memory makes this trivial — Husk knows what state the session was in.

## 16. Open questions — RESOLVED (2026-05-25)

The seven questions below have been decided. Encoded here for reference; do not re-litigate during Phase A implementation.

**Locked decisions:**

1. **State-graph authoring** — **Agent-authored only (fully autonomous).** No hand-authored graphs. Every site is learned from the ground up via exploration. First agent on any site pays cold-start (~30-60s exploration). Subsequent agents inherit. Forces exploration mode to be excellent on day 1.

2. **LLM access in cognition layer** — **LLM-neutral. Deterministic predicates only.** Confidence + regexes + AX-role matching + network observation. No LLM dependency. Decision A holds.

3. **Default trust for new sites** — **Conservative: read-only until graph matures.** Destructive intentions (post, pay, delete) blocked until graph reaches confidence > 0.85 OR user explicitly raises trust.

4. **Stale-graph recovery** — **Confidence decay + on-demand re-exploration.** Confidence decays slowly over time (~1% / week). Below threshold → next use of that transition triggers re-exploration. Background calibration validates top sites periodically.

5. **Community sharing of state graphs** — **Opt-in, redacted by default.** Graphs stay local unless user explicitly publishes. When published, structural data only — URLs, query params, anything with potential PII gets redacted by the publisher tool. Per-site granular opt-in. No registry in v0.1 itself; data model leaves room for v0.2.

6. **v0 tool deprecation** — **Two-release deprecation, remove in v0.3.** v0.1 ships the 8 new tools alongside the existing 21; v0 tools marked deprecated. v0.2 logs warnings. v0.3 removes them. Anyone with v0 integrations has the v0.1 release cycle to migrate.

7. **Concurrent exploration coordination** — **First-explorer-wins; others wait or use partial graph.** Per-site exploration lock prevents duplicate work. Agents starting mid-exploration on the same site read the partial graph; subsequent intentions either wait for completion or proceed with low-confidence execution.

**Original questions for archival context:**

### 16.1 State-graph authoring — who writes them?

Three models:

- **Hand-authored** by Husk maintainers + community: high quality, narrow coverage
- **Agent-authored**: the first agent to visit a site builds the graph automatically; high coverage, variable quality
- **LLM-authored offline**: an LLM consumes a site's snapshots + transitions and writes the YAML; medium quality, scalable

Best answer is probably hybrid: hand-author for the top 50, agent-author for the long tail, LLM-distill periodically.

### 16.2 Schema versioning

State graphs evolve. LinkedIn ships a new UI; the graph is out of date. How does Husk detect and recover? Versioning + per-version confidence + automatic re-exploration when confidence drops below threshold.

### 16.3 Adversarial sites

If sites realize Husk exists, they'll fingerprint the engine, detect the state-machine patterns, and break us deliberately. We're not the first to face this (cf. Browser-Use, Playwright); the answer is the standard stealth toolkit. But the state-machine layer is a NEW fingerprint vector — sites could detect "this client navigates with the precision of a state machine" and block.

Mitigation: add stochastic timing, slight randomization of click positions, etc. Built into the cognition layer, not the agent's problem.

### 16.4 Failure when state graph is wrong

If the state graph says "submit_login → home_feed" but the site is now broken and goes to a different page, the cognition layer needs to gracefully degrade. Probably: drop confidence, switch to exploration mode for this intention, alert the user.

### 16.5 Privacy of community state graphs

If state graphs include URL patterns / form field names, they're not personal data. But if memory includes session history, that's sensitive. Hard split between "site model" (shareable) and "agent history" (private).

### 16.6 LLM API access — necessary or no?

Tempting to use an LLM internally for "is this state a captcha?" / "is this content an error?" / "is this button the primary action?" Currently Husk is LLM-neutral (Decision A). v0.1 might benefit from an *optional* LLM API hook, but the core path must stay LLM-neutral. Confidence-driven exploration is the alternative.

### 16.7 What's the right initial trust setting for a brand-new site?

Conservative: every action requires user confirm until proven. Permissive: act freely; user pulls back if needed. Default needs to be conservative; let the user dial it up.

### 16.8 Backwards compatibility

How long do we keep the 21 v0 tools around? Forever in `husk_advanced`? Deprecated for a release cycle then dropped? Probably: forever — the v0 surface is the lower-level escape hatch. Power users will want it. Most agents won't.

### 16.9 Pricing / hosting model

If the community state-graph registry exists, who pays? Static hosting of YAML is cheap (~free on a static-site CDN). Verification + signing might cost more. Probably: free for personal use, sponsored or freemium for commercial.

### 16.10 The "Husk Hub"

Sketch: a hosted dashboard where users can:
- Browse community state graphs and playbooks
- Publish their own (with optional anonymization)
- See aggregate stats ("LinkedIn-connect-flow worked 92% of the time this week")
- Audit their own agent's actions (replay, redact, share)

This is a separate product. Don't scope creep v0.1 into building it. But the data model in v0.1 should leave room for it.

## 17. What v0.1 is not

A few things we explicitly defer to v0.2 or later:

- **Voice integration** — sometimes the user wants to talk, not type. Defer.
- **Multi-machine orchestration** — Husk on your laptop + Husk on a server + Husk on your phone. Defer.
- **Cross-org playbook sharing with access control** — commercial feature, defer.
- **Visual debugging** — a "rewind the agent's perspective frame-by-frame" view. Defer.
- **The Husk Hub itself** — separate product, separate spec.
- **Browser fingerprinting evasion ("stealth mode")** — important but its own spec, defer.
- **Agent personality / style transfer** — "write LinkedIn DMs in my voice". Out of scope for the browser.

## 18. Success criteria

How we'll know v0.1 worked.

### 18.1 The LinkedIn test

> "Send connection requests to these 9 people on LinkedIn."

Today (v0 + patches): 9 tool calls, 2-3 manual interventions, ~50% success rate. Agent gives up sometimes.

v0.1 target: **1 tool call, 0 manual interventions, 95% success rate**. Total wall clock ~60s (most of it rate-limit pauses).

### 18.2 The cold-site test

> "Find the SOC 2 report for [obscure SaaS vendor]'s trust page and download it."

Today: doesn't work. Agent doesn't know the site, falls over.

v0.1 target: agent + cognition layer explores the site, builds a state graph in 30-60s, executes the intention. State graph is then cached for the next agent.

### 18.3 The handoff test

> "Pay this invoice on Stripe, but stop and ask me before actually submitting payment."

Today: agent tries, hits an unexpected modal, asks for help, frustrating loop.

v0.1 target: agent runs the playbook, Husk's destructive-action gating pauses BEFORE submit, asks user "Confirm $X charge to Y? yes/no", user confirms, payment goes through. 1 tool call from agent, 1 user touch.

### 18.4 The replay test

> "Repeat what you did last Tuesday on LinkedIn."

Today: doesn't work. No memory.

v0.1 target: agent invokes the audit log, finds last Tuesday's session, replays with `husk_play({from_audit: <session_id>})`. Works.

## 19. Closing — why this matters

Husk v0 is a competent browser engine for AI agents. It works. It does what's on the box.

But it asks the agent to do the wrong things — translate intentions to clicks, parse AX trees, verify outcomes by inspection. That's how a fast human would use a browser. It's not how an agent should.

Husk v0.1 inverts the relationship. **The browser becomes the model of the web.** The agent reasons about intentions. The browser handles everything else.

This unlocks:
- Workflows that today require 20-tool-call loops collapse to 1
- Sites that today break the agent (LinkedIn, Gmail, modern SaaS) become first-class
- Knowledge accumulates: every agent benefits from every prior agent's success
- The agent's context budget goes to reasoning, not parsing
- Real automation, not assisted clicking

The cost is real: 6-8 weeks of focused work. The shift is architectural, not incremental. But every patch we've shipped since M14 is a tax on the wrong architecture — and the tax is increasing.

If we believe agents are how people will use the web in five years, the question isn't whether to build v0.1. It's whether to start before someone else does.

## Appendix D — Glossary

- **Affordance** — an action available in the current state
- **Capability** — a feature the agent's task needs (e.g., `modern_javascript`)
